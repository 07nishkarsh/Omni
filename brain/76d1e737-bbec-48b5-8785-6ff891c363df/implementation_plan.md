# Implement Orchestrator Loop and Negotiation

The goal is to tie Agents A, B, and C together into a robust state machine that drives the full transaction lifecycle and handles multi-round negotiations.

## Proposed Changes

### `app/orchestrator/state_machine.py`
[MODIFY] [state_machine.py](file:///Users/nishkarsh/Omni/Codes/app/orchestrator/state_machine.py)
- Refactor `StateMachine` to execute the workflow:
  - **Review Stage**: Invoke Agent A to get the routing decision (`A->B->C` or `A->C`).
  - **Execution Stage**: Invoke Agent B (if in route) and Agent C sequentially.
  - Convert their output into a `Proposal` object.
  - If Agent B vetoes, `status = REJECTED`.
  - If Agent C returns `PARTIAL`, `status = COUNTERED` (triggering negotiation).
  - If Agent C returns `TREASURY_REJECT`, `status = REJECTED`.
  - If Agent C returns `APPROVED`, `status = APPROVED`.
- Catch `LLMUnavailableError` and safely escalate to `TransactionStatus.ESCALATED`.

### `app/orchestrator/negotiation.py`
[MODIFY] [negotiation.py](file:///Users/nishkarsh/Omni/Codes/app/orchestrator/negotiation.py)
- Create a `negotiate_transaction(ctx: TransactionContext) -> list[Proposal]` function (or similar class).
- Implement a loop (max 4 rounds):
  1. Call the pipeline (`state_machine.py` logic) to get a `Proposal`.
  2. Log the proposal to an in-memory transcript list.
  3. If `Proposal.status == APPROVED` or `REJECTED` or `ESCALATED`, consensus/termination is reached. Break loop.
  4. If `Proposal.status == COUNTERED` (e.g. Agent C returned `PARTIAL` with a lower `availableAmount`):
     - Update the `TransactionContext` (e.g., reduce `requested_amount` to `availableAmount` to simulate a disbursement split counter-proposal).
     - Continue to the next round.
  5. If the loop reaches 4 rounds without consensus, create a final `Proposal` that escalates to human review (Manager Approval Desk).
- Validate the entire proposal history using `validator.validate_proposal_history(transcript)` at the end.
